# DDM Field Attribute LocalService Extension

Liferay DXP OSGi modules that expose Liferay's internal
`com.liferay.dynamic.data.mapping.model.DDMFieldAttribute` model through a
custom local service. Liferay's dynamic-data-mapping module persists DDM field
attributes but does not publish a local service for them — this project fills
that gap with a Service Builder–generated `DDMFieldAttributeLocalService`
whose implementation delegates to Liferay's own `DDMFieldAttributePersistence`.

A small test portlet is included for exercising the service: it fetches a
`DDMFieldAttribute` by ID, lets you edit its attribute value, and saves it.

## Modules

| Module     | Artifact                                | Description                                                                                             |
|------------|-----------------------------------------|---------------------------------------------------------------------------------------------------------|
| `api/`     | `com.sohlman.ddmfieldattribute.api`     | API bundle: the `DDMFieldAttributeLocalService` interface and supporting classes generated by Service Builder. |
| `service/` | `com.sohlman.ddmfieldattribute.service` | Service bundle: `DDMFieldAttributeLocalServiceImpl` (the custom logic) plus `service.xml`.               |
| `portlet/` | `com.sohlman.ddmfieldattribute.portlet` | `DDM Field Attribute Test` MVC portlet (display category *Sample*) for manual testing.                   |
| `deploy/`  | `ddmfieldattribute-deploy`              | Assembly that packages the three bundles into `osgi/modules/` as a directory, `.zip`, and `.tar.gz`.     |

## Requirements

- JDK 17 (an `.envrc` is provided that points `JAVA_HOME` at an SDKMAN Zulu 17 install)
- No Maven install needed — the Maven Wrapper (`./mvnw`, Maven 3.9.16) is included
- Liferay DXP 2025.Q1 (built against BOM `2025.q1.15`, see `liferay.bom.version` in the root `pom.xml`)

## Building

The project is not inside a Liferay workspace, so activate the `standalone`
profile, which imports the Liferay BOMs and adds the Liferay repositories:

```bash
./mvnw clean install -Pstandalone
```

Note when bumping `liferay.bom.version`: not every `2025.qX.Y` patch is
published to the public Liferay repository. Verify that `release.dxp.bom`,
`release.dxp.bom.compile.only`, and `release.dxp.bom.third.party` all exist
for the target version before changing it.

## Service Builder

`service/service.xml` is the source of truth for the service layer. After
changing it, regenerate from the `service/` module:

```bash
cd service
../mvnw service-builder:build -Pstandalone
```

Generated code is written into `api/src/main/java` and
`service/src/main/java`. Custom logic belongs only in
`service/src/main/java/com/sohlman/ddmfieldattribute/service/impl/DDMFieldAttributeLocalServiceImpl.java`
— other generated files are overwritten on regeneration. Re-run Service
Builder after adding methods to the `*Impl` class so the interfaces are
updated.

## Deploying

After a build, the deploy module produces ready-to-ship packages under
`deploy/target/`:

- `ddmfieldattribute-deploy-<version>/` (directory layout)
- `ddmfieldattribute-deploy-<version>.zip`
- `ddmfieldattribute-deploy-<version>.tar.gz`

Each contains the three bundles under `osgi/modules/`. Either unpack the
archive over your Liferay home (so the jars land in `LIFERAY_HOME/osgi/modules`)
or copy the jars into `LIFERAY_HOME/deploy`.

## Trying it out

1. Deploy the bundles and add the **DDM Field Attribute Test** portlet
   (display category *Sample*) to a page.
2. Pass a `fieldAttributeId` request parameter (the primary key of a row in
   Liferay's `DDMFieldAttribute` table) to load an attribute.
3. Edit the attribute value in the form and save — the portlet updates the row
   through `DDMFieldAttributeLocalService`.

## Caveats

`DDMFieldAttribute` is an internal Liferay model, and this service writes to
Liferay's own `DDMFieldAttribute` table, bypassing any higher-level DDM
business logic. Use with care — primarily intended for testing, migrations,
and repair tooling rather than as a general-purpose API.
